# M23.1 — Что такое макросы

**Длительность:** ~14 минут
**Тема:** препроцессор, директива `#define`, макросы как замена текста

## Главная идея

> Препроцессор — отдельный «слой языка» поверх C++.
> `#define` создаёт **макрос** — текстовую подстановку **до компиляции**.

## Контекст: что вообще делает компилятор

```text
исходный .cpp файл
       ↓
   ПРЕПРОЦЕССОР  ← здесь работают #include, #define
       ↓
   КОМПИЛЯТОР    ← здесь работает C++ как язык
       ↓
   ЛИНКЕР
       ↓
   .exe / .bin
```

Препроцессор оперирует **текстом**, а не типами. Его директивы:
- `#include "file"` — подставить содержимое файла
- `#define NAME value` — заменить `NAME` → `value` в коде
- `#ifdef`, `#ifndef`, `#endif`, `#pragma`, `#error`, ...

Все начинаются с `#`.

## #include: уже знакомый макрос

```cpp
#include <iostream>
```

Препроцессор берёт файл `iostream` и **дословно вклеивает** его текст
на место директивы. Результат — одна большая «склеенная» строка кода.

## #define: новая директива

```cpp
#define TITLE "Skillbox"
```

**Синтаксис:**
- `#` — начало директивы
- `define` — команда
- `TITLE` — идентификатор макроса (без типа!)
- `"Skillbox"` — значение (всё, что справа до конца строки)

**Конвенция:** имена макросов — `UPPER_SNAKE_CASE`.

### Без `=` и без типа

```cpp
#define TITLE "Skillbox"      // НЕТ знака равно
#define MAX_USERS 100         // НЕТ типа
```

Макрос — не переменная. Он не занимает память, не имеет типа.
Просто инструкция препроцессору: «везде, где видишь TITLE, подставь "Skillbox"».

## Использование

```cpp
#include <iostream>
#define TITLE "Skillbox"

int main() {
    std::cout << TITLE;   // выводит: Skillbox
}
```

После препроцессинга компилятор видит:
```cpp
int main() {
    std::cout << "Skillbox";
}
```

## Конкатенация строковых литералов

```cpp
#define TITLE "Skillbox"

const char* phrase = "The " TITLE " Online Courses";
//                   ↑^^^^^^   ^^^^^   ^^^^^^^^^^^^
// Три литерала рядом → C++ автоматически склеивает:
// → "The Skillbox Online Courses"
```

**Почему работает:** в C/C++ соседние строковые литералы конкатенируются
на этапе компиляции. После раскрытия макроса получаем три литерала рядом.

### Не работает с обычной переменной

```cpp
const char* TITLE = "Skillbox";
const char* phrase = "The " TITLE " ...";   // COMPILE ERROR
// Литерал и переменная нельзя соединить пробелом
```

Только **литералы** склеиваются. Переменная не литерал.

## Макросы как константы

```cpp
#define UP    1
#define DOWN  2
#define LEFT  3
#define RIGHT 4

if (cmd == UP)   { /* ... */ }
if (cmd == DOWN) { /* ... */ }
```

Препроцессор подставит:
```cpp
if (cmd == 1) { /* ... */ }
if (cmd == 2) { /* ... */ }
```

### vs `enum` — что лучше?

| Аспект                | #define    | enum                  |
|-----------------------|------------|-----------------------|
| Тип                   | нет        | enum-тип              |
| Type safety           | нет        | да                    |
| Виден в отладчике     | редко      | да                    |
| Scope                 | глобальный | enum class — скоупный |
| Препроцессор          | да         | нет                   |

> **Учитель рекомендует `enum`** для типобезопасности.
> `#define` оставлять для случаев, требующих именно подстановки текста.

## Битовые маски через макросы

```cpp
#define UP    (1 << 0)   // 0001 = 1
#define DOWN  (1 << 1)   // 0010 = 2
#define LEFT  (1 << 2)   // 0100 = 4
#define RIGHT (1 << 3)   // 1000 = 8

#define UP_RIGHT   (UP | RIGHT)     // 1001 = 9
#define UP_LEFT    (UP | LEFT)      // 0101 = 5
#define DOWN_RIGHT (DOWN | RIGHT)   // 1010 = 10
#define DOWN_LEFT  (DOWN | LEFT)    // 0110 = 6
```

```cpp
if (dir == UP_RIGHT) {
    // движение по диагонали вправо-вверх
}
```

## ВАЖНО: скобки вокруг значения

```cpp
#define BAD  1 << 0       // без скобок
int x = BAD * 2;          // → 1 << 0 * 2 → 1 << 0 = 1 (приоритет!)

#define GOOD (1 << 0)     // со скобками
int x = GOOD * 2;         // → (1 << 0) * 2 → 1 * 2 = 2
```

> **Правило:** макросы с выражениями ВСЕГДА оборачивай в скобки.
> Препроцессор подставляет текст «как есть», и операторы могут
> склеиться с окружающим кодом неожиданным образом.

## Опасности макросов

- **Нет типа** — компилятор не проверит совместимость
- **Нет скоупа** — макрос виден везде, можно случайно «затереть» переменную
- **Дебагер не видит** — в стектрейсе будет уже raw value
- **Подстановка текста** — могут быть сюрпризы с приоритетами
- **`#define MAX(a,b) ((a) > (b) ? (a) : (b))`** — двойное вычисление аргументов!

## Practical relevance для DevOps

- **`#ifdef DEBUG`** — условная компиляция (debug vs release builds)
- **`#define VERSION "1.2.3"`** — встроить версию в бинарь
- **`#define BUFFER_SIZE 1024`** — конфигурация на этапе компиляции
- **Cross-platform:** `#ifdef _WIN32 ... #else ... #endif`
- **Build flags:** `gcc -DDEBUG=1 -DVERSION=\"1.2.3\"` — define из командной строки

## Связь с другими модулями

- **M1+ #include:** уже использовали препроцессор для подключения header'ов
- **M16.5 enum:** альтернатива #define для константных значений
- **M23.2+ (next):** функциональные макросы, условная компиляция, header guards
- **C/C++ ABI:** макросы — ключ к платформенно-зависимому коду
