# M15.3 — Оптимизация задач на массивах

**Длительность:** 15:12
**Источник:** `L3_Оптимизация_задач.vtt` / `.txt`
**Тема:** O(n²) → O(n) — single-pass с трекингом «текущего минимума»

## Главная идея модуля

> «Алгоритм должен не только давать правильный ответ, но и работать оптимально.»

Первый раз показывают **алгоритмическую сложность** — пусть и без формальной
big-O нотации. Учитель оперирует «миллион×миллион vs несколько миллионов»,
что отлично передаёт интуицию квадратичного vs линейного времени.

## Задача 1 — Best Time to Buy/Sell Stock (one transaction)

**Вход:** `vector<int> prices = {7, 1, 5, 3, 6, 4}`
**Выход:** `5` (купить за 1 в день 2, продать за 6 в день 5)

### Наивная версия — O(n²)

```cpp
int max_pr = 0;
for (int i = 0; i < prices.size(); i++) {
    for (int j = i + 1; j < prices.size(); j++) {
        if (prices[j] - prices[i] > max_pr) {
            max_pr = prices[j] - prices[i];
        }
    }
}
```

Двойной цикл: для каждого «дня покупки» проверяем все «дни продажи» после него.
Для миллиона дней — триллион операций.

### Оптимизация — O(n), single-pass

```cpp
int min_v = prices[0];
int max_pr = 0;
for (int i = 1; i < prices.size(); i++) {
    if (prices[i] < min_v) min_v = prices[i];
    if (prices[i] - min_v > max_pr) max_pr = prices[i] - min_v;
}
```

**Ключевое наблюдение:** в каждой точке нужен только минимум **слева от неё**.
Если в текущий день цена выше минимума — продали бы при том минимуме.
Если ниже — это новый кандидат на минимум для будущих продаж.

**От триллиона до нескольких миллионов** для миллиона дней.

## Задача 2 — Best Time to Buy/Sell Stock (unlimited transactions)

**Вход:** тот же массив
**Выход:** `6` (купить день2 за 1 → продать день3 за 5; купить день4 за 4 → продать день5 за 6)

### Решение — O(n), один проход

```cpp
int profit = 0;
for (int i = 1; i < prices.size(); i++) {
    if (prices[i] - prices[i - 1] > 0) {
        profit += prices[i] - prices[i - 1];
    }
}
```

**Идея:** «держим акции только в дни роста». Сумма всех положительных
разниц подряд = максимально возможная прибыль при unlimited transactions.

Доказательство: любая «горка» от низа до верха = сумма всех инкрементов
по пути → собирая каждый положительный delta, получаем максимум.

## Концепции, которые впервые вводятся

### 1. Алгоритмическая сложность (без формальной big-O)

Учитель называет это «количество операций» и сравнивает:
- O(n²) = миллион × миллион = триллион
- O(n) = «несколько миллионов»
- O(1) per element = «нижняя граница, быстрее не получится»

Это первый намёк на тему complexity которую формально дадут позже.

### 2. Single-pass + state-tracking

Паттерн: вместо двух циклов — один проход с **аккумулятором состояния**:
- `min_v` (минимум слева)
- `max_pr` (текущий максимум прибыли)

Этот же паттерн потом всплывёт в:
- Maximum subarray (Kadane's algorithm)
- Sliding window
- Two pointers

### 3. «Оптимизация под обоснование»

Учитель явно говорит: «быстрее одного прохода не получится — нам всё равно
надо прочитать данные». Это идея **lower bound** — некоторые задачи имеют
теоретический минимум времени, и линейный алгоритм его достигает.

### 4. Greedy thinking

Задача 2 — пример **greedy** (жадного) подхода:
- Локально: каждый положительный delta = profit
- Глобально: сумма всех = глобальный optimum
- Доказательство через «сложение разниц по горке»

Без формального введения греди — но идея «принимай локально оптимальное
решение, и оно даст глобально оптимальное» уже звучит.

## Edge cases (не покрыто в видео)

### Задача 1 (one transaction)
- Цены монотонно убывают `{5, 4, 3, 2, 1}` → `max_pr = 0` (ничего не покупаем)
- Один день `{5}` → `max_pr = 0`
- Все равные `{5, 5, 5}` → `max_pr = 0`
- Пустой вектор → `prices[0]` UB! (валидация на boundary)

### Задача 2 (unlimited transactions)
- Монотонно убывающий → `profit = 0`
- Монотонно растущий `{1, 2, 3, 4, 5}` → `profit = 4` (купить день1, держать до конца — то же что суммировать deltas)
- Один день → `profit = 0` (цикл с i=1 не запускается)

Эти edge cases надо покрыть в hwork-тестах.
