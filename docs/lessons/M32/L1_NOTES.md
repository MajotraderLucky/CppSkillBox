**Длительность:** ~8 минут

## Главная идея

> **JSON** = JavaScript Object Notation — стандартный текстовый формат для
> хранения и передачи **структурированных данных**.
>
> Идеален для **сериализации классов** в C++: ключи JSON ↔ имена полей класса.

## Контекст: сериализация

> **Сериализация** = превращение объекта в **поток байт** для сохранения / передачи.
> **Десериализация** = обратное: восстановление объекта из потока.

Раньше изучали:
- Текстовое чтение/запись (M19/M20) — **неструктурированное**
- Бинарное I/O — **сырые байты**

Теперь — **структурированный** формат → JSON.

## Происхождение JSON

| Часть | Расшифровка     |
|-------|-----------------|
| `JS`  | JavaScript      |
| `ON`  | Object Notation |

- Изначально — фрагмент языка JavaScript (фронтенд)
- Стал стандартом индустрии для **обмена данными** между сервисами / языками

## Пример: ответ httpbin.org/get

```json
{
    "args": {},
    "headers": {
        "Accept": "*/*",
        "Host": "httpbin.org",
        "User-Agent": "curl/7.74.0"
    },
    "origin": "1.2.3.4",
    "url": "http://httpbin.org/get"
}
```

## Структура JSON

### Базовый блок — словарь (`{...}`)

- **Фигурные скобки** — границы объекта (как в C++ блоки кода)
- Внутри — пары `"ключ": значение`, разделённые запятыми
- **Ключи всегда строки** (в двойных кавычках)
- Ключи **уникальны** в пределах одного словаря

> Семантика — как `std::map<std::string, JsonValue>`.

### C-подобные языки

> Фигурные скобки `{}` пришли из языка C → распространились в C++/Java/JavaScript/C#.
> Все они называются **C-like languages**.

Если знаешь C++ — легко прочитать код на этих языках.

### Связь с классами C++

| JSON                     | C++                              |
|--------------------------|----------------------------------|
| Ключ `"name"`            | Имя поля класса                  |
| Значение                 | Значение поля                    |
| Вложенный `{}`           | Член-класс / композиция          |
| Массив `[...]`           | std::vector / array              |

> **Сериализация:** обходим поля → пишем `"имя": значение` → JSON-строка.
> **Десериализация:** читаем JSON → ищем ключ → присваиваем полю.

## Типы значений в JSON

| Тип               | Пример                   | C++ аналог                   |
|-------------------|--------------------------|------------------------------|
| Строка            | `"hello"`                | `std::string`                |
| Число             | `42`, `3.14`, `-0.5`     | `int` / `double`             |
| Логическое        | `true`, `false`          | `bool`                       |
| Null              | `null`                   | `nullptr` / отсутствие       |
| Объект (словарь)  | `{"k": "v"}`             | struct / map                 |
| Массив            | `[1, "x", null]`         | vector (heterogeneous!)      |

> **null** — «нет значения». Аналог `nullptr`. Записывается без кавычек.

### Два способа задания «пусто»

```json
"args": {}        // пустой словарь
"args": null      // явное «нет значения»
```

Оба валидны, но семантика **разная**:
- `{}` — словарь существует, но без полей
- `null` — словаря нет

## Работа в IDE (CLion)

1. `File → New Scratch File`
2. Тип: **JSON** (или JSON5 — расширенная версия, но менее распространена)
3. Вставить контент → подсветка синтаксиса
4. Ошибки структуры (запятые, скобки) подсвечиваются автоматически

> JSON5 backwards-compatible с JSON, но не везде поддерживается → используем классический JSON.

## Валидация структуры

Минимальный валидный JSON:
```json
null
```
или
```json
{}
```
или
```json
{"key": "value"}
```

**Невалидно:**
- Trailing comma: `{"a": 1,}`  — после последнего поля **нельзя** запятой
- Single quotes: `{'a': 1}`     — только двойные
- Без quotes у ключа: `{a: 1}`  — ключи всегда в кавычках
- Comments: `{"a": 1 /* x */}`  — JSON **не поддерживает** комментарии (а JSON5 — да)

## Зачем JSON в DevOps / C++

- **REST API:** ответы сервисов → JSON
- **Конфиги:** AWS, GCP, kubernetes manifests (часто YAML, но JSON под капотом)
- **Логи:** structured logging (JSON lines)
- **Telemetry/metrics:** Prometheus, OpenTelemetry — JSON exporters
- **CI/CD:** GitHub Actions / GitLab pipelines — JSON outputs
- **package.json / composer.json / etc.** — manifests

## Сравнение: JSON vs альтернативы

| Формат   | Плюсы                              | Минусы                           |
|----------|------------------------------------|----------------------------------|
| **JSON** | Стандарт, простой, поддержан везде | Нет комментариев, verbose        |
| YAML     | Читабельный, комментарии           | Indentation-sensitive, сложнее   |
| TOML     | Чёткие типы, конфиги               | Менее распространён              |
| XML      | Validatable, namespaces            | Verbose, медленный парсинг       |
| Protobuf | Бинарный, быстрый, schema          | Не human-readable                |

## Practical relevance для DevOps

- **HTTP API:** REST = JSON in/out (как видели в M30 с CPR)
- **Парсинг JSON в C++:** `nlohmann/json` (header-only, идиоматичный)
- **Configuration as code:** JSON конфиги → C++ daemon
- **Logging:** structured logs → ELK / Loki / Datadog
- **Schema validation:** JSON Schema → проверка корректности
