# M34.2 — Отрисовка и архитектура GUI (Event Loop)

**Длительность:** ~14 минут
**Тема:** растровая графика, framebuffer как 2D массив, оконные системы, **Event Loop**

## Главная идея

> **Рисование** = заполнение пикселей в `framebuffer` цветами.
>
> **GUI** = иерархия окон/виджетов, управляемая **циклом обработки событий** (Event Loop).
> События = `EventLoop → getMessage → dispatch → handler`.

## Растровая vs векторная графика

| Тип                | Что хранит                | Артефакты при увеличении | Где используется        |
|--------------------|---------------------------|--------------------------|-------------------------|
| Растровая (raster) | Массив пикселей           | Лесенки, "квадратики"    | Paint, JPEG, PNG, экран |
| Векторная          | Геометрия (линии, кривые) | Нет (масштабируется)     | SVG, AI, PDF, шрифты    |

> Экран монитора — **растровый**. Поэтому даже векторное изображение в итоге **растеризуется** для отображения.

> В Paint видно артефакт растра при zoom — линия превращается в "лесенку".

## Framebuffer как 2D массив

```cpp
const int W = 1920;
const int H = 1080;
Color framebuffer[H][W];   // 2D массив строк
```

> Под капотом — **1D массив**, индексируется как `framebuffer[y * W + x]`. Двумерная нотация — синтаксический сахар.

## Система координат буфера

```text
(0, 0) ─────────► X
   │
   │
   │
   ▼ Y
```

> **Y отсчитывается СВЕРХУ ВНИЗ** — стандарт почти всех оконных систем (отличается от математической XY).

## Простейшее рисование

### Точка

```cpp
framebuffer[40][100] = {255, 0, 0};   // красная точка x=100 y=40
```

### Горизонтальная линия

```cpp
Color blue {0, 0, 255};
for (int x = 100; x <= 800; ++x) {
    framebuffer[40][x] = blue;
}
```

### Прямоугольник (заполненный)

```cpp
for (int y = 40; y <= 1000; ++y) {
    for (int x = 100; x <= 800; ++x) {
        framebuffer[y][x] = blue;
    }
}
```

### Принцип

> Любая фигура (кнопка, скроллбар, иконка) — **заполнение пикселей** в правильных координатах.
>
> GPU обычно рисует **треугольниками** — даже сложные формы разбиваются на треугольники.

## Оконные системы (Windowing System)

> Иерархия окон → внутри окна интерфейс → внутри интерфейса другие виджеты.

### Что делает оконная система

- **Отрисовка** графических элементов
- **Обработка пользовательского ввода** (клавиатура, мышь)
- **Менеджмент фокуса** (какой виджет получает события)
- **Маршрутизация событий** (какому окну передать клик)

### Активное окно

> Только **активное окно** получает event'ы клавиатуры. Кликнул в другое окно — фокус сменился.

## Event Loop (цикл обработки событий)

### Аналогия с консольной программой

```cpp
// Консольная программа в M21+:
while (true) {
    std::string cmd;
    std::cin >> cmd;
    if (cmd == "exit") break;
    if (cmd == "add")    handleAdd();
    if (cmd == "remove") handleRemove();
}
```

### GUI программа (псевдокод)

```cpp
while (true) {
    Message msg = getMessage();    // блокирующее ожидание следующего event'а
    if (msg.type == EXIT) break;
    dispatchMessage(msg);          // направить event обработчику
}
```

> Концептуально **то же самое**: бесконечный цикл, читающий команды и направляющий их обработчикам.

### Функции

| Функция                 | Назначение                                      |
|-------------------------|-------------------------------------------------|
| `getMessage()`          | Получить следующий event из очереди (или ждать) |
| `dispatchMessage(msg)`  | Перенаправить event соответствующему виджету    |

### Dispatch — это `if/switch` встроенный в систему

```cpp
// Что концептуально делает dispatchMessage:
switch (msg.type) {
    case MOUSE_CLICK: routeToWidgetUnderCursor(msg);    break;
    case KEY_PRESS:   sendToFocusedWidget(msg);         break;
    case RESIZE:      resizeWindow(msg);                break;
    // ...
}
```

> Это **тот же самый паттерн**, что мы писали в консольных hwork — только встроенный в систему.

## Структура Message (event)

```cpp
struct Message {
    int  type;     // тип события (enum / int)
    char key;      // для KEY_PRESS
    int  x, y;     // для MOUSE_MOVE
    // ...
};
```

### Поля

| Поле       | Назначение                                  |
|------------|---------------------------------------------|
| `type`     | Идентификатор типа (MOUSE/KEY/RESIZE/...)   |
| `key`      | Какая клавиша нажата                        |
| `x, y`     | Координаты курсора                          |

> Можно использовать **enum** (экономия) или **string** (читаемость).

## Главная опасность: блокировка Event Loop

### Что происходит при долгой операции

```cpp
void onButtonClick() {
    for (int i = 0; i < 1'000'000'000; ++i) {  // тяжёлая работа
        compute(i);
    }
}
```

> Пока `onButtonClick` выполняется → `getMessage()` **не вызывается** → events **накапливаются** в очереди → программа **«не отвечает»**.

### Симптом

> «Application not responding». Курсор зависает. Пользователь не может ничего нажать.

### Решение: **отдельный поток** для тяжёлой работы

```cpp
void onButtonClick() {
    std::thread([] {
        for (int i = 0; i < 1'000'000'000; ++i) compute(i);
    }).detach();
}
```

> **Главный поток** = Event Loop, должен оставаться **отзывчивым**.
> **Worker threads** = долгие операции, IO, вычисления.

## Системные сообщения

Сообщения бывают не только от пользователя:

| Категория   | Примеры                              |
|-------------|--------------------------------------|
| User input  | KEY_PRESS, MOUSE_MOVE, MOUSE_CLICK   |
| Window mgmt | RESIZE, MINIMIZE, CLOSE, FOCUS_LOST  |
| System      | COPY, PASTE, UNDO, REDO              |
| App-level   | TIMER_TICK, NETWORK_DATA, IPC        |

> Event Loop — **универсальный** механизм обработки **любых** асинхронных сигналов.

## Practical relevance для DevOps

- **Event-driven architecture:** message bus / queues используют тот же паттерн
- **Async runtimes:** Node.js, Python asyncio, Rust tokio — все на event loop
- **Web servers:** nginx, envoy — event loop поверх epoll/kqueue
- **GUI freezes:** диагностика — найти блокирующий код в main thread, вынести в worker
- **Profilers:** Chrome DevTools "Long Tasks", instruments в Xcode — ищут блокирующие callbacks

## Почему важна модель

> «Несмотря на видимую простоту, event loop **достаточен** для самых сложных интерфейсов.»

Все крупные UI-фреймворки (Qt, WPF, GTK, Cocoa, Web) — реализация одного и того же паттерна:
1. Очередь событий
2. Цикл `getMessage` → `dispatch` → `handle`
3. Worker threads для тяжёлых задач
