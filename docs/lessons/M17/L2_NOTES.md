# M17.2 — Определение размера переменных + арифметика указателей

**Длительность:** ~12 минут
**Тема:** `sizeof`, арифметика указателей, доступ к массиву через `*(arr+i)`

## Главная идея

> Арифметика над указателями работает в **единицах размера типа**, а не в байтах.

## sizeof

```cpp
int a = 5;
std::cout << sizeof(a);     // 4 байта
std::cout << sizeof(int);   // 4 байта
std::cout << sizeof(double); // 8 байт
```

Принимает имя переменной ИЛИ имя типа.

### Размер указателя — одинаковый!

```cpp
int* pi;
double* pd;
std::cout << sizeof(pi);    // 4 (или 8 на 64-bit)
std::cout << sizeof(pd);    // 4 (или 8 на 64-bit)
```

**Все указатели имеют одинаковый размер**, потому что они хранят
адреса в одном адресном пространстве.

## Массив = указатель на первый элемент

```cpp
int r[5] = {1, 2, 3, 4, 5};
std::cout << r;       // адрес r[0] (hex)
std::cout << *r;      // 1 (значение r[0])
```

Имя массива неявно конвертируется в указатель на первый элемент.

## Арифметика указателей

```cpp
int r[5] = {1, 2, 3, 4, 5};
int* p1 = r + 1;    // указатель на r[1]
int* p3 = r + 3;    // указатель на r[3]
std::cout << *p1;   // 2
std::cout << *p3;   // 4
```

**Ключ:** `r + 1` означает «следующий элемент», а не «следующий байт».
Компилятор сам умножает на `sizeof(*p)`:
- Для `int*`: +1 = +4 байта (на 32-bit int)
- Для `double*`: +1 = +8 байт

Можно увидеть в hex: разница между `r` и `r+1` для `int[]` = 4, для `double[]` = 8.

## Итерация массива через указатели

```cpp
for (int i = 0; i < 5; i++) {
    *(r + i) *= 2;          // эквивалент r[i] *= 2
}
for (int i = 0; i < 5; i++) {
    std::cout << *(r + i) << " ";
}
// Output: 2 4 6 8 10
```

Эквивалент `r[i]` = `*(r + i)`. Квадратные скобки — синтаксический сахар над арифметикой указателей.

## Опасность: выход за границы

```cpp
int r[5] = {...};
int* p = r + 10;     // компилируется! но указывает на foreign memory
std::cout << *p;     // мусор, или segfault, или undefined behavior
*p = 42;             // КАТАСТРОФА: затираем чужую память
```

**Segmentation fault** возникает при доступе к защищённой памяти.
Знание `sizeof` критично — иначе можно прочитать «не до конца» (получить
часть числа) или «слишком много» (зайти в чужую область).

## Practical relevance для DevOps

- **C arrays as buffers:** `char buf[256]; read(fd, buf, sizeof(buf));`
- **Stack buffer overflow vulnerabilities:** классика — strcpy без bounds check
- **Network buffers:** sizeof для определения payload size
- **Memory mapping:** mmap возвращает void*, нужно cast + sizeof для arithmetic
- **gdb:** `x/16xw $rsp` — сдвиг в bytes/words/halfwords/...
