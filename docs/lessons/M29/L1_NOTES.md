# M29.1 — Инкапсуляция (1-й принцип ООП)

**Длительность:** ~7 минут
**Тема:** парадигмы программирования, **инкапсуляция** как замкнутая система
**Преподаватель:** Денис Крахмалёв (новый)

## Главная идея

> Любая программная сущность с **нетривиальным состоянием** должна быть
> **замкнутой системой** — переход только из одного **корректного** состояния
> в другое.

## Парадигмы программирования (исторически)

| Парадигма          | Идея                                                     | Время          |
|--------------------|----------------------------------------------------------|----------------|
| Imperative         | Последовательность команд                                | Машинный код   |
| Structured         | Циклы, условия, функции                                  | C, Pascal      |
| **Procedural**     | Программа = большая процедура (вход → обработка → выход) | C 70-80        |
| **OOP**            | Объекты с состоянием + поведением                        | C++/Java/...   |
| Functional         | Чистые функции, immutable                                | Haskell, Scala |
| Logic              | Декларативный («что», а не «как»)                        | Prolog         |

> **Procedural** — наши прошлые модули (большая `main` функция).
> **OOP** — следующая ступень: данные + поведение в одном классе.

## Принципы ООП (4 основных)

1. **Инкапсуляция** ← этот урок
2. **Наследование**
3. **Полиморфизм**
4. (иногда) **Абстракция**

## Что такое инкапсуляция

> **Инкапсулировать** = "запаковать в капсулу", скрыть внутренности.

**Не только скрытие данных, но и контроль изменений.**

## Проблема: открытое состояние

```cpp
class Dog {
public:
    int age;     // открытое поле
};

Dog d;
d.age = 10;     // OK
d.age = -50;    // тоже компилируется! Логическая ошибка
d.age = 5000;   // тоже OK с точки зрения компилятора
```

Класс не контролирует свои инварианты. Любой код снаружи может «сломать» собаку.

## Решение: private + конструктор + getter/setter

```cpp
class Dog {
    int age;     // private (default для class)

public:
    Dog(int age) {
        if (age < 0 || age > 30) {
            std::cout << "Incorrect age\n";
            return;
        }
        this->age = age;
    }

    void setAge(int age) {
        if (age < 0 || age > 30) {
            std::cout << "Incorrect age\n";
            return;
        }
        this->age = age;
    }

    int getAge() const {
        return age;
    }
};

Dog d(10);          // конструктор валидирует
d.setAge(11);       // setter валидирует
std::cout << d.getAge();   // безопасное чтение
```

### Что инкапсуляция даёт

- **Валидация в одной точке** — все изменения через setter
- **Невозможно создать объект в плохом состоянии** — конструктор проверяет
- **Внутренние представления можно менять** — снаружи API стабильно
- **Расширение поведения** — добавили лог в setter → все changes логируются

## Не только скрытие, но и **управление**

> Цель инкапсуляции — не **запретить**, а **дать управлять корректным образом**.

Без getter/setter класс был бы бесполезен.
С только getter — read-only.
С setter — изменения, но **только валидные**.

## Пример: что бывает без инкапсуляции

```cpp
struct BankAccount {
    double balance;
};

BankAccount acc;
acc.balance = -1000000;     // отрицательный баланс? UB
```

С инкапсуляцией:

```cpp
class BankAccount {
    double balance = 0;
public:
    void withdraw(double amount) {
        if (amount <= 0)         { /* error */ return; }
        if (amount > balance)    { /* insufficient funds */ return; }
        balance -= amount;
    }
    void deposit(double amount) {
        if (amount <= 0)         { /* error */ return; }
        balance += amount;
    }
    double getBalance() const { return balance; }
};
```

## Naming conventions для getter/setter

| Стиль         | Пример                         | Где встречается |
|---------------|--------------------------------|-----------------|
| `getX/setX`   | `getAge/setAge`                | Java, C# style  |
| `x()/setX(v)` | `age()/setAge(v)`              | Qt, Boost       |
| `x()/x(v)`    | `age()/age(v)`                 | C++ современный |
| `x_/x()`      | private `age_`, public `age()` | Google style    |

## Когда НЕ инкапсулировать

- **POD структуры** (Plain Old Data) — просто данные, нет инвариантов
- **Internal data classes** (детали реализации)
- **Когда state тривиален** — `struct Point { double x, y; }`

> **Правило:** если есть **инвариант** между полями, или валидация — инкапсулировать.

## Practical relevance для DevOps

- **API design:** только public методы стабильны; private — можно менять
- **Refactoring safety:** если поле private — менять можно безопасно
- **Audit/logging:** все изменения через setter → центральная точка
- **Validation:** не разбросано по коду, в одном месте
- **Thread safety:** mutex inside setter, не expose raw data
