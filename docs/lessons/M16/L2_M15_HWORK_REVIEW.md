## Сравнение моих реализаций с эталоном учителя

### Задача 1 — Maximum Subarray

| Аспект         | Эталон учителя          | Моё решение            |
|----------------|-------------------------|------------------------|
| Алгоритм       | Naive O(n³)             | Kadane O(n)            |
| Helper         | `sum_elements(from,to)` | inline cur_sum/start   |
| Output         | `Sum: X i: Y j: Z`      | `i j` (только indices) |

Эталон использует тройной вложенный цикл с helper-функцией суммирования.
Мой Kadane быстрее (O(n) vs O(n³)) и достаточно простой. Логика
обновления `min_index, max_index` идентична.

### Задача 2 — Two Sum

| Аспект         | Эталон                           | Моё                     |
|----------------|----------------------------------|-------------------------|
| Алгоритм       | O(n²) + early `return`           | O(n²) + early-exit      |
| Inner loop     | `j = i` (allows i == j)          | `j = i + 1` (no i == j) |
| Output         | `First: X` `Second: Y` (2 lines) | `X Y` (1 line)          |

Учитель допускает `j = i` — число суммируется с самим собой если `2*a[i] = target`.
В реальной задаче «два числа в массиве» это пограничный случай — может быть и так и так.
Моя реализация строже (требует разные индексы).

### Задача 3 — Top-5 smallest

| Аспект         | Эталон                         | Моё                 |
|----------------|--------------------------------|---------------------|
| Алгоритм       | swap-based вставка             | insert+pop_back     |
| Trigger        | `vec.size()<5 \                | \                   | val<vec[4]` | size<K else replace if val<back |
| Print          | print vector after each insert | print only on -1    |

Учитель печатает массив **после каждой вставки** (debug output?).
Я печатаю только при запросе -1. Учитель аккуратнее с проверкой
наличия 5 элементов («Num doesnt exist»). Мой вариант идентичен по логике
(top-5 maintain), просто другой способ insert (swap vs insert).

### Задача 4 — Sort by absolute value

| Аспект         | Эталон                            | Моё                              |
|----------------|-----------------------------------|----------------------------------|
| Алгоритм       | two-pointer (find border + merge) | identical                        |
| Compare        | `vec[right] > -vec[left]`         | `abs(left) < abs(right)` (equiv) |
| Tail loops     | 2 separate loops в конце          | 1 loop с early-exit checks       |

**Логически идентично.** Учитель отделяет main merge от tail вывода
двумя отдельными while. Я объединил в один цикл с проверками `left < 0`
/ `right >= n`. Equivalent в плане complexity и output.

## Ключевые выводы

1. **Учитель ожидает naive решения** — это для intro-курса, не для оптимизации.
   Мои оптимизированные решения (Kadane) превосходят, но важно понимать что
   spec не требует optimal.

2. **Output format не стандартизирован** — я использовал минималистичный
   («числа через пробел»), учитель — verbose («Sum: X i: Y j: Z»).
   Для submission лучше следовать учительскому стилю.

3. **Edge cases у учителя НЕ тестируются** — нет валидации, нет empty
   array handling. Мои реализации более устойчивы.

4. **`using namespace std`** в Task 3 — учитель использует, что не
   рекомендуется в production, но допустимо в учебном коде.
